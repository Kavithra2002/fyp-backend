TECHNICAL EXPERT 1 — Senior Software Engineer
--------------------------------------------------------------------------------
Subject: Review notes — API design and service boundaries

Dear Kavithra,

Thank you for sharing your FYP backend for review. Overall, the separation between
the Node API and the Python ML service is a sensible choice: it keeps inference
and training concerns out of the request path for the core web layer and allows
you to scale or swap the ML component independently.

I was glad to see explicit input validation on auth and forecast-related endpoints.
That is exactly where many student projects fail first under malformed input.
My main suggestion is to document the contract between the two services very
clearly—request/response schemas, error codes, and timeouts—and to add integration
tests that exercise failure modes (ML service down, slow response, invalid model
output shape). That will strengthen both your report and your viva when asked
about reliability.

Second, consider stating your strategy for idempotency and observability: if a
client retries a forecast request, do you deduplicate or accept duplicates, and
how do you log correlation IDs across Node and Python?

Regards,
Senior Software Engineer

Subject: Feedback on ML service integration and reproducibility

Dear Kavithra,

I reviewed the description of your ML microservice and training workflow. The
direction—serving models from a dedicated service with a controlled debug flag and
optional path restrictions—is aligned with basic production hygiene.

For a stronger technical narrative, I recommend you spell out how model versions
are tracked and how training data lineage is preserved (even a lightweight
approach: dataset hash, training notebook version, and serialized model filename
in your API metadata). Reviewers will ask how you prevent silent drift between
what was evaluated offline and what is deployed.

Also clarify evaluation metrics used during training versus what the API exposes
to end users (point forecasts only vs. intervals, confidence, etc.). If explain
endpoints surface feature attributions, briefly justify the method and its
limitations for time-series or tabular data as applicable.

TECHNICAL EXPERT 3 — Senior Software Engineer

Subject: Security review — authentication and API hardening

Dear Kavithra,

Thank you for the walkthrough of your security baseline. Mandatory JWT secret,
HttpOnly cookie support alongside bearer tokens, helmet, rate limiting, and CORS
allowlisting are all appropriate controls for a student API that will be
demonstrated and possibly deployed.

A few concrete improvements to consider for your documentation and implementation:
(1) Rotate and never commit secrets—ensure .env.example documents variables
without real values; (2) describe your threat model briefly (who attacks what);
(3) if file or data paths reach the ML layer, confirm path traversal and prefix
checks are enforced consistently; (4) rate limits should be justified with
expected legitimate traffic vs. abuse.

These points will read well in a viva when examiners ask “what if someone tries
to break this?”

Kind regards,
Senior Software Engineer

Dear Kavithra,

The feature set—authentication, data routes, forecasting, and explainability—

Subject: Practical feedback — will this help day-to-day decisions?

Dear Kavithra,

From an operations angle, forecasting tools only matter if they connect to how
we actually plan—stock, promotions, staffing, and supplier lead times. Your focus
on a structured API suggests you are thinking about integration rather than a
one-off spreadsheet, which is promising.

I would push you to state clearly which business decisions your forecasts support

make about seasonality and promotions. If those assumptions are wrong, the model
can look accurate on paper but mislead buyers.

A short “limitations for real stores” section in your report will show maturity.

Best,

DOMAIN EXPERT 2 — Senior Software Engineer
--------------------------------------------------------------------------------
Subject: Data and metrics — what should stakeholders trust?

Dear Kavithra,

Analysts care about definitions: what is a “sale,” how returns are handled,
whether revenue is gross or net, and what time granularity you forecast (daily,
weekly). Without that, two teams will interpret the same chart differently.

Please document your target metric and horizon, and how you handle missing or
late data—common in real e-commerce pipelines. If your system exposes
explanations, clarify whether they are for model debugging or for business
storytelling; those are related but not identical.

This will make your work more credible to domain reviewers.

Kind regards,
Senior Software Engineer

Subject: Responsible analytics in commercial forecasting

Dear Kavithra,

Forecasting systems can influence purchasing and pricing decisions; even when
automated, they should be interrogated for bias and transparency. If your data
reflects historical inequities or promotional distortions, the model can encode
them.

I encourage a brief discussion of data governance: consent, retention, who can
access forecasts, and how wrong predictions would be detected in production.
You do not need a legal essay—just evidence that you considered downstream impact.

This complements your technical security discussion nicely.
